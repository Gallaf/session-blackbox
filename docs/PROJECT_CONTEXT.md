# Project Context

Session Blackbox is an experimental module for Foundry Virtual Tabletop.

The project owner is a senior C# developer with experience in backend systems,
architecture, SQL, and DevOps, while still learning JavaScript, TypeScript, and
the Foundry VTT ecosystem. Project documentation and implementation should be
clear, incremental, and explicit about decisions. Avoid magical abstractions.

## Objective

The module should behave like a lightweight black box for a game session.

When chat messages and rolls are created in Foundry, Session Blackbox should
capture a compact representation of them for future export and post-session
analysis.

A future workflow may combine this Foundry log with a timestamped transcript of
the session. That combined data could support:

- session recaps;
- timelines;
- important event extraction;
- rules questions;
- narrative analysis;
- post-game review.

## Core Boundary

AI analysis must not happen inside Foundry.

During the session, the module must not call AI, external APIs, external
network services, or anything that could make gameplay depend on outside
systems.

During play, Session Blackbox should only:

- observe;
- copy compact data;
- keep that data locally;
- stay out of the way.

## Central Principle

Store the log and be invisible in performance.

"Invisible in performance" means the module should not:

- cause lag;
- block or delay rolls;
- do heavy work inside hooks;
- pollute the UI;
- perform I/O for every message;
- scan `game.messages`;
- resolve Actor, Item, Token, or Scene documents through expensive lookups;
- parse DOM or rendered HTML inside hooks.

## Transparency

The module should be transparent to GMs and players. Everyone should be able to
know that capture is active.

The initial MVP will not implement UI or a visual notice yet. Transparency is a
project requirement for a later step, not something to fake in code before the
basic capture path exists.

## Desired Compatibility

The target compatibility is Foundry VTT v13 and v14.

Do not claim compatibility until the module has been tested in those versions.

## Known Local Environment

- Foundry dev data path: `C:/FoundryDataDev`
- Preferred local repository: `C:/Repos/session-blackbox`
- Expected Foundry module path:
  `C:/FoundryDataDev/Data/modules/session-blackbox`
- Local strategy: use a junction from the Foundry module path to the repository.

## Current Architectural Decision

Session Blackbox must coexist with modules such as Chat Cleaner.

Chat Cleaner may delete old messages from the visual chat, but Session Blackbox
should capture the compact message data before cleanup happens.

Expected flow:

1. A `ChatMessage` is created in Foundry.
2. Session Blackbox captures a compact copy through a hook.
3. The message continues normally through Foundry.
4. Another module, such as Chat Cleaner, may delete it later.
5. Session Blackbox still has the compact local copy for future export.

## Hooks Considered

### Main Hook

`createChatMessage`

This should be the primary capture point because it observes the created
`ChatMessage` document. It can include regular messages, system messages,
cards, rolls, and messages generated by systems or modules.

### Future Fallback

`preDeleteChatMessage`

This may later help preserve minimal metadata before deletion, but it is not
part of the first MVP.

### Future Optional Hook

`deleteChatMessage`

This may later help understand cleanup behavior, but it is not part of the
first MVP.

### Hooks Not To Use As The Primary Source

`chatMessage`

This hook is closer to text submitted through the chat log. It is not the best
primary source for system-generated messages, roll cards, or module-created
messages.

`renderChatMessageHTML`

This is a rendering/UI hook. It is more fragile and potentially more expensive
because it deals with rendered output rather than the created document.

## Privacy Policy

Whispers, blind rolls, and private messages require extreme care.

Safe default behavior:

- do not save `content` for whispers, blind rolls, or private messages;
- do not save `flavor` for whispers, blind rolls, or private messages;
- save only minimal metadata for private entries;
- set a `contentOmitted` indicator when content was intentionally omitted.

The module must not capture private content by default.

## Strong Restrictions

The MVP must not:

- send data to AI;
- send data to the network;
- export automatically;
- save private content without explicit configuration;
- deep-clone full `flags` data;
- store full `ChatMessage` objects;
- parse DOM or rendered HTML;
- scan existing message history;
- do heavy document resolution;
- add UI;
- add settings;
- add sockets;
- add build tooling;
- add TypeScript;
- add npm dependencies.

## Implementation Style

The initial implementation should use:

- simple JavaScript;
- no build step;
- no mandatory npm setup;
- no TypeScript in the first MVP;
- small files;
- short comments only where they genuinely help;
- explicit performance and privacy decisions.
